So what's the issue here? Let's look at the markup of that first button.
We can see that the button is an anchor ("a") element, without a href attribute.
Without the attribute, the anchor can't receive focus, as the browser doesn't consider this an interactive element.
A screenreader user would also have to deal with this issue, so how can we fix that?
Its really pretty trivial.
We can add a placeholder href attribute to an anchor.
And we can make a span focusable by adding the tabindex="0"
Better yet though, we use an actual button element for a button, and an actual anchor for links.
We can still layer behavior we JavaScript on top, but starting with proper markup is essential.

Once you’ve got a screenreader running, which is at most a matter of running the installer and then starting the installed application, you need to know a little bit about navigation with a screenreader. Unlike your previous keyboard accessibility testing, screenreader users don’t use the Tab key as much. Instead they use other methods of navigation. Pretty much all of those have shortcuts mapped to single keys, so for example in JAWS you press H to go to the next header. There’s another shortcut to go through links (anchors). You can also use the cursor keys to move your virtual cursor. Starting from a header, cursor down will move the virtual cursor to the next element, so for example the screenreader would then read the text of the paragraph below a header.
