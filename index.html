[play video, narrate a bit]
Search works, but selecting the actual flight fails as can't focus that area (at least there's no visual focus).
So what's the issue here? Let's look at the markup of that first button.
We can see that the button is an anchor element, but without an href attribute.
There's probably JavaScript attached to it, doing something on click.
But without the attribute, the anchor can't receive focus, because the browser doesn't consider this an interactive element.
So we can't trigger that click event with the keyboard.
A screenreader user would also have to deal with this issue. How can we fix that?
Its really pretty trivial.
We can add a placeholder href attribute to an anchor. And we can make a span focusable by adding the tabindex attribute
Even better would to use an actual button element for a button, and an actual anchor for links.
We can still add custom behavior with JavaScript, but starting with proper markup is essential.
Let's check out two more examples of bad markup...

Markup for links and buttons
The product title here should really be a header.
The product image needs an alt attribute with a useful description.
The decoration image needs an alt attribute with an empty value. That tells screenreaders to ignore this image.
Do this whenever there is no description for an image. Don't skip the alt attribute, otherwise the screenreader reads the image filename. Which usually sounds pretty terrible.
One more example...

Markup for content
I found this table on another airline site. Don't do that.
Link labels to inputs using the for-attribute. Otherwise a screenreader won't know how to label the input. We've seen that happening on the amazon site as well, where we heard "edit text", but not "search edit text".
If all these solutions seem trivial: They are, but all too often they aren't implemented.

All desktop screenreaders have something in common, and while they use different names, I'll just call it "Virtual Cursor".
The virtual cursor is one way of navigating documents and the primary way to read documents.
It basically allows you to walk through the DOM tree, reading one element after the other.

The two major aspects that ARIA consists of are roles...[first bullet]
We've seen landmarks in action before. Other roles are widget specific, we'll take a look at those soon
ARIA also specifies varios states and properties [second bullet]. We've just used aria-live and aria-atomic, two of these properties.
As any web standard, ARIA isn't without problems. Sometimes we have to use workarounds instead of just strictly following the spec to make things work...
