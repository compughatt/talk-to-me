How to test?

Start with the keyboard

 Now that we have a basic idea of what people are using on the internet, how can we test our sites? Start with keyboard accessibility. Its easy to check, just put away your mouse (or touchpad) and use only the keyboard. Use the Tab key to navigate your page. Try to interact with everything that is interactive, usually links and forms. If there’s something else, like a lightbox image gallery, check that: It should have cursor left/right bindings for showing the next and previous images, and escape key for closing the overlay. Look for issues like a >span> that is styled like a link. It looks like a link, you could click on it, but it turns out that you can focus it with tabbing.
This video shows the Wizzair site, an airline. I'm looking for a flight, using only the keyboard.
Search works, but selecting the actual flight fails as can't focus that area (at least there's no visual focus).
So what's the issue here? Let's look at the markup of that first button.
We can see that the button is an anchor element, without a href attribute.
Without the attribute, the anchor can't receive focus, as the browser doesn't consider this an interactive element.
Adding the href or tabindex=0 and just making it a button would help.
That’s the type of problem a screenreader user would also have to deal with, so we can eliminate a bunch of issues from that category before even testing with a screenreader. Though eventually we need to do that, too. So what should we test with, and how?
You should start with NVDA+Firefox on Windows. That’s free to use, easy to install and has a pretty decent marketshare.
With a Mac available, you should enable VoiceOver in System Preferences and check with Safari (no need to install anything; no other browser works properly with VoiceOver).
JAWS has the biggest market share, you can test the demo version for 40 minutes before having to restart your computer. In that regard, NVDA is a lot cheaper. Window-Eyes is also pretty popular, but also commercial with only a 30-minute trial before requiring a restart. Also worth mentioning is VoiceOver on iOS. Its very different from the OSX version. You should test with it especially if you’re building a mobile focused site.

What screenreaders to test?

  • start with NVDA with Firefox on Windows
  • on OSX, enable VoiceOver and test with Safari
  • to be thorough, test JAWS with IE8+
  • to be really thorough, test Window-Eyes with IE8+

If you’re on Linux or OSX, using a VM works. You can even get good Windows VMs from the modern.IE project. Once you’ve got a screenreader running, which is at most a matter of running the installer and then starting the installed application, you need to know a little bit about navigation with a screenreader. Unlike your previous keyboard accessibility testing, screenreader users don’t use the Tab key as much. Instead they use other methods of navigation. Pretty much all of those have shortcuts mapped to single keys, so for example in JAWS you press H to go to the next header. There’s another shortcut to go through links (anchors). You can also use the cursor keys to move your virtual cursor. Starting from a header, cursor down will move the virtual cursor to the next element, so for example the screenreader would then read the text of the paragraph below a header. Let’s see how that works by looking at this video of me navigating Google Search with NVDA: [video] Now we know how to test our site ourselves, from really low effort keyboard testing to more involved screenreader testing. But how do we fix issues when we find them? Remember Progressive Enhancement? That’s still pretty useful. When you start with decent markup, you get a lof of default interactions/behavior for free. A button can be “clicked” with Enter and Space keys, a link can be clicked with the Enter key. No need for custom JS on a span element. Though if you really need to make something tabbable that isn’t already, use tabindex=”0”. The zero tells the browser to add the element inside the regular tab order. If you’re recreating a button, you need to then recreate all the native key bindings, which tends to be rather messy. jQuery doesn’t help you there, since it attempts no normalization of key events. Similar to using >button> or >a>, use the right header element for headers, then style those. Don’t just make a >div> look like a header. Quick aside about colors: Good contrast is important! Tools can help with that and even check colors against common issues with color blindness ( http://www.checkmycolours.com/ ). I’m sure you’ve heard before that you have to add alt attributes to images. That still applies, but only add text that’s actually useful. If there’s nothing useful to say and the image should be ignored by the screenreader, use alt=””. Don’t leave it out, since then the screenreader may end up reading the filename of the image. So that’s the easy stuff. What if we are working on a widget that has no representation in HTML? Isn’t that what ARIA is all about? Isn’t that kind of not working so well? Yep, ARIA can be pretty terrible. Back to our >span> link, we can use role=”link” to tell the screenreader what we’re pretending here. It gets a little more complicated though. Generally its a good idea to use a framework that is already tested against screenreaders and makes use of the right ARIA roles and other attributes. One such framework is jQuery UI. Its no coincidence that I’m working on jQuery UI and giving this talk, since I’ve spent a good amount of time making jQuery UI widget’s accessible, including a lot of screenreader testing.
